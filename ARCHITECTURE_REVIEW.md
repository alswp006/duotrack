# Architecture Review
Generated: 2026-07-22T16:49:52.903Z
Score: 80/100

## Summary
This is a small, well-organized React/TypeScript mini-app (App-in-Toss) with a clear layered structure: presentational 'pre-built' components in src/components, domain types/storage/api in src/lib, and thin page components in src/pages. The Result-object error pattern (ok/code, ok/reason) is used consistently in lib/api.ts and lib/storage.ts, and there are no circular dependencies. The main issues are small pockets of copy-pasted SDK-guard and numeric-text-wrapper logic, a couple of likely-dead exports, and one file that mixes a dev-only artifact into the production pages folder.

## Issues
### Warnings
- [duplication] The 'fire haptic feedback with a try/catch because the SDK throws outside the Toss WebView' pattern is implemented twice independently: as a local fireHaptic() helper in BottomCTA.tsx and re-inlined verbatim inside FloatingTabBar.tsx's onClick handler. Any future change to haptic error handling (e.g. adding logging) must be made in two places and will drift. — Files: src/components/BottomCTA.tsx, src/components/FloatingTabBar.tsx / Fix: Extract a shared `fireHaptic(type)` utility (e.g. src/lib/haptics.ts) that both components import, instead of BottomCTA owning a private copy and FloatingTabBar re-implementing the same try/catch + Promise.resolve(...).catch(()=>{}) logic.
- [duplication] Amount.tsx and CountUp.tsx both wrap their output in an identical `<span>` with the same inline style object (display: inline-block, maxWidth: 100%, whiteSpace: nowrap, fontVariantNumeric: tabular-nums) and the same formatNumber+unit rendering, differing only in the value source (static vs animated). This is a byte-for-byte duplicated presentation layer. — Files: src/components/Amount.tsx, src/components/CountUp.tsx / Fix: Extract a shared internal `NumericText` wrapper (or a shared style constant) that both Amount and CountUp use for the nowrap/tabular-nums span, so the visual contract lives in one place.
- [dead_code] `formatCurrency` is exported from lib/utils.ts but no shown component or page calls it — Amount.tsx and CountUp.tsx both use formatNumber with a manually appended unit string instead, suggesting formatCurrency is unused/orphaned. — Files: src/lib/utils.ts / Fix: Remove formatCurrency if truly unused, or adopt it in Amount/CountUp if currency-style formatting was actually intended, to avoid two competing number-formatting entry points.
- [naming] types.ts declares both `export const GoalConfig = Symbol('GoalConfig')` and `export interface GoalConfig` under the exact same identifier. This is a legal but confusing TypeScript pattern (value/type merging) and the symbol has no visible consumer in the given files, making it look like leftover test scaffolding masquerading as a domain type. — Files: src/lib/types.ts / Fix: Rename the runtime marker (e.g. `GoalConfigMarker`) to avoid shadowing the type name, or remove it if the 'test introspection' use case is no longer needed.

### Suggestions
- [duplication] The 'probe SDK support via isSupported?.(), swallow the throw outside WebView, and no-op' guard pattern is independently re-implemented in AdSlot.tsx (isBannerSupported/ensureInitialized), TossPurchase.tsx, and TossRewardAd.tsx, each with its own try/catch shape and comments explaining the same WebView-vs-browser rationale. — Files: src/components/AdSlot.tsx, src/components/TossPurchase.tsx, src/components/TossRewardAd.tsx / Fix: Introduce a shared `safeSdkCall`/`guardOutsideWebView` helper in src/lib that centralizes the try/catch-and-fallback logic, and have each component call it instead of hand-rolling the guard three times.
- [dead_code] `OPIC_GRADE_ORDINAL` and `OpicGrade` are exported from types.ts but are not referenced by storage.ts, api.ts, Home.tsx, or NotFound.tsx in this codebase slice — no OPIC grade comparison/sorting logic is wired up anywhere visible. — Files: src/lib/types.ts / Fix: Confirm a consumer exists elsewhere in the app; if not, drop it until the OPIC scoring feature is actually built, to keep the type module from accumulating speculative exports.
- [structure] src/pages/__TdsGallery.tsx is a dev-only component-catalog page living in the same folder as real routed pages (Home.tsx, NotFound.tsx). It's tree-shaken at build time, but its presence in src/pages makes the folder's contents ambiguous ('is this a real page?') and couples dev tooling to the production page directory. — Files: src/pages/__TdsGallery.tsx, src/App.tsx / Fix: Move dev-only scaffolding pages into a dedicated folder (e.g. src/dev/TdsGallery.tsx) and import from there in App.tsx, keeping src/pages exclusively for real routes.
- [dead_code] Several 'pre-built, do-not-reimplement' components (MiniBar, Sparkline, AdSlot, TossPurchase, TossRewardAd, FloatingTabBar, StateView's LoadingState) are not imported by any page shown (Home.tsx and NotFound.tsx only use ScreenScaffold, SummaryHero, Card, EmptyState). This may be intentional design-system scaffolding for future pages, but as of this snapshot they are unreferenced exports. — Files: src/components/MiniBar.tsx, src/components/Sparkline.tsx, src/components/AdSlot.tsx, src/components/TossPurchase.tsx, src/components/TossRewardAd.tsx, src/components/FloatingTabBar.tsx / Fix: If these are meant as a shared library for upcoming pages, consider a README or barrel export noting that; otherwise verify usage build-wide with a dead-export lint (e.g. ts-prune) before shipping.
- [consistency] Error handling for SDK/network failures is inconsistent in observability: lib/api.ts and lib/storage.ts surface typed Result objects ({ok,...}) that callers can branch on, while the SDK-wrapper components (AdSlot, BottomCTA's fireHaptic, FloatingTabBar's inline haptic call) silently swallow all failures with empty catch blocks and no caller-visible signal at all. — Files: src/lib/api.ts, src/lib/storage.ts, src/components/AdSlot.tsx, src/components/BottomCTA.tsx / Fix: This may be an intentional UX choice (haptics/ads are non-critical), but document the convention explicitly (e.g. in a CONTRIBUTING note) so it reads as a deliberate policy rather than an accidental inconsistency versus the Result-typed lib layer.
